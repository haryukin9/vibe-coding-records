# 記事テンプレート

## 標準構成（5章）

各記事は以下の5章構成を基本とする。記事の性質によって章の比重は変えてよいが、章立ての順序は変えない。

### 1. 業務背景（なぜ作る必要があったか）

- どんな現場の、どんな課題か
- 既存のやり方の問題点（工数、コスト、ヒューマンエラー、属人化など）
- 「これがしんどかった」という実感ベースで書く
- **数字を入れる**（週◯時間、月◯件、年間◯万円など）
- 著者の業務範囲・規模感もここで匂わせる

### 2. ディレクション設計（AIに何を、どう伝えたか）

- このシリーズの**最大の差別化ポイント**。必ず厚めに書く
- AIにいきなり実装を頼むのではなく、要件定義をどう言語化したか
- 曖昧な要望を具体化するために何を考えたか
- プロンプト全文や、最初の指示書の抜粋を載せると良い
- 「映像ディレクションでも同じ筋肉を使う」など、過去の経験との接続を語る（ただし業種は伏せる）

### 3. 実装プロセス（どうやって作ったか・どこでハマったか）

- 使ったツール（Claude Code、◯◯API、ライブラリ名など）を具体的に
- 開発の流れを時系列で
- ハマったポイント・失敗・回り道を必ず1つは入れる（読み手の親近感）
- AIとの対話の抜粋を1～2回分入れると臨場感が出る
- ステップごとのスクリーンショット（Playwrightキャプチャ）を入れる

### 4. 成果物デモ（何ができたか）

- 完成したものの全体像
- スクリーンショット or SVG図解で見せる
- 主要機能を3つ程度に絞って解説
- 動かしている様子（GIFや短い動画リンク）があれば最強

### 5. 学び・転用先（次に何ができるか）

- 今回得た学びを抽象化して書く
- 「同じ仕組みは◯◯にも使える」「次はこれを作りたい」など発展性を提示
- 採用担当が「この人、他でも応用できそう」と感じる流れを作る
- シリーズ次回予告や関連記事へのリンクで締める

## タイトルの書式

```
【ひとりDX担当の記録】<具体的な題材>を<動詞>た話
```

例：
- 【ひとりDX担当の記録】会社のWebサイトを全部AIで作り直した話
- 【ひとりDX担当の記録】PageSpeed Insightsを満点に近づけたブラッシュアップ全工程
- 【ひとりDX担当の記録】SVGメーカーで画像作成コストをゼロにした

冒頭の「【ひとりDX担当の記録】」は固定。シリーズ感を出す。

## 文字数

- **本文 3000～5000字**（noteの推奨レンジ）
- FAQは設けない（noteは検索流入よりタイムライン流入が主）
- 長くなりすぎる場合は前後編に分ける

## 画像の方針

- アイキャッチ：必ず1枚。SVGでシリーズ統一デザインが理想（後で固定テンプレ化する）
- 本文中：1～3章ごとに1枚、最低2～3枚
- スクリーンショット：実画面を見せる場合は機密情報をマスクする
- 図解：SVGで作成→PNG変換→GitHub Pages経由でnote埋め込み

## 文体ルール

- 「である調」ではなく「ですます調」
- 「弊社」「我が社」は使わず「うちの会社」「自社」
- 段落は短め（3～4行）
- 重要な数字は太字
- リストや見出しを多めに使う（noteは流し読み読者が多い）
