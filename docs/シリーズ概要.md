# シリーズ概要

## シリーズ名（冠）

**「中小企業のひとりDX担当が、AIで会社をアップデートする」**

各記事タイトルの冒頭に必ずこのシリーズ名（または短縮形「ひとりDX担当の記録」）を入れるか、ハッシュタグで補足する。

## 著者の立ち位置

- 中小企業に所属する**ひとりDX担当**として発信する
- 業種・業界・社名は明示しない（業界バレを避ける）
- 「クリエイティブ業界の中小企業」「ひとりで複数業務を回している」までは出してよい
- 自分が**非エンジニア**であることは前面に出す
- 「コードは読めない／書けない」「AIに依頼してものを作っている」スタンスを徹底

## 想定読者

優先順：

1. **事業会社のDX/AI推進担当者・経営層**（採用側）
2. 中小企業の業務担当者・ひとり情シス的ポジションの人
3. 「Claude Code・バイブコーディングを試してみたい」非エンジニア
4. 副次的に、AI活用に興味のある人全般

## トーン

- **一人称・実体験ベース・カジュアル寄り**
- 専門用語は使わない／使うときは必ず日本語で言い換え
- 「やってみた」「ハマった」「結果こうなった」の体験談スタイル
- 失敗・試行錯誤も隠さず書く（読み手の信頼を得る）
- 過剰な自慢・煽り表現は避ける（「神ツール」「革命」等は使わない）

## 差別化ポジション

「非エンジニア×Claude Code」発信は2026年時点で**完全な激戦区**。
このシリーズが取るポジションは以下：

- **業務インフラを内製した実録**（個人開発・MVPレベルではなく、現場稼働している）
- **AIへの依頼の言語化術**（曖昧な要望をAIに伝える設計力）
- **小さい規模だからこその意思決定スピード**（数十人未満の会社で全部ひとりで決められる）

避けるポジション：
- 「3日で◎◎を作った」系のスピード自慢
- 「全員使うべき」系の総論
- 副業・収益化系

## 記事の役割

各記事は**ポートフォリオの1ページ**として機能させる。
- 採用担当が見たときに、職務経歴書のキーワードと一致する語彙を使う
  - 例：「内製化」「業務インフラ」「DX」「自動化」「工数削減」「AI活用」
- 数字で語れるものは数字で語る（工数◯時間→◯時間、SaaSコスト◯万円→0円、など）
- 業務背景を必ず添える（「なぜ作る必要があったか」）
